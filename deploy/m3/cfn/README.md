# BYOC substrate (`m3-byoc-substrate.yaml`)

One CloudFormation template that provisions the customer-cloud substrate the
Gentrail Helm chart deploys onto: VPC, EKS, the 6 DynamoDB tables, RDS Postgres,
a KMS CMK, S3 (evidence + logs + trace archive), the trace-archiver Lambda, and
per-ServiceAccount IRSA roles. The same chart runs on k3s (Gentrail cloud) and
here; only the values differ, generated by `../scripts/cfn-to-values.sh`. Trace
retention matches Gentrail cloud: traces expire from DynamoDB after 30 days
(TTL) and the archiver Lambda copies them to the trace-archive bucket, which
tiers to GLACIER_IR at 90 days and DEEP_ARCHIVE at one year.

## Deploy

The template is >51 KB, so it cannot be passed inline; stage it in S3 first.
The trace-archiver Lambda zip is staged the same way: build it from
`services/ingestion/lambda-archiver` (`CGO_ENABLED=0 GOOS=linux GOARCH=arm64
go build -o bootstrap . && zip archiver.zip bootstrap`) and upload it to a
bucket in the stack's region.

```bash
aws s3 cp m3-byoc-substrate.yaml s3://<bucket>/cfn/m3-byoc-substrate.yaml
aws s3 cp archiver.zip s3://<bucket>/cfn/archiver.zip
aws cloudformation create-stack --stack-name gentrail-substrate \
  --template-url https://s3.<region>.amazonaws.com/<bucket>/cfn/m3-byoc-substrate.yaml \
  --capabilities CAPABILITY_NAMED_IAM \
  --parameters ParameterKey=Hostname,ParameterValue=app.acme.com \
               ParameterKey=ExternalId,ParameterValue=<cross-account-external-id> \
               ParameterKey=ArchiverCodeS3Bucket,ParameterValue=<bucket> \
               ParameterKey=ArchiverCodeS3Key,ParameterValue=cfn/archiver.zip

# then, once CREATE_COMPLETE:
aws eks update-kubeconfig --name <ClusterName>
../scripts/cfn-to-values.sh gentrail-substrate > values.byoc.yaml
helm install gentrail ../helm/gentrail -f values.byoc.yaml --set image.tag=<tag>
```

For the validation/test loop, `cfn-deploy.sh` and `cfn-teardown.sh` automate this
end to end (env-overridable; defaults are a minimal sandbox install):

```bash
./cfn-deploy.sh     # stage template, create-stack, wait, then deploy the chart
./cfn-teardown.sh   # uninstall, drop RDS (no snapshot), delete-stack, sweep orphans
```

`cfn-teardown.sh` is deliberately more than `delete-stack`: it skips the RDS final
snapshot and sweeps the `DeletionPolicy: Retain` resources (see Teardown notes),
which a plain stack delete leaves behind.

## Prerequisites the template cannot grant for you

- **Bedrock model access is a manual, per-account-per-region console step.** AWS
  gates foundation models (Anthropic Claude included) behind an account opt-in on
  the Bedrock "Model access" page; neither CloudFormation nor IAM can enable it.
  A fresh account often has only `amazon.nova-*` enabled. The `BedrockModelId`
  CustomResource is therefore **advisory**: if the model is not yet enabled it
  emits a WARNING (visible on the `BedrockModelValidation` resource) and the
  stack still completes. The policy engine is the only Bedrock consumer; enable
  Claude access before turning it on (`policyEngine.replicaCount > 0`).
- **EKS 1.32 has no in-tree EBS provisioner.** The stack installs the EBS CSI
  addon, but you must create a CSI-backed default `StorageClass`
  (`provisioner: ebs.csi.aws.com`) for the chart's PVCs to bind. Set
  `--set storage.class=<that class>`.
- **ALB ingress** requires the AWS Load Balancer Controller installed in-cluster
  and a DNS record pointed at the provisioned ALB (created post-helm-install).
  `ingress.certArn` is only needed when TLS terminates at the ALB; it is empty
  on the no-public-cert path (no `HostedZoneId`) and the chart accepts that.

## Teardown notes

- **RDS takes a final snapshot on delete** (default `Snapshot` deletion policy).
  This requires the instance to be in `available` state, so deleting a stack that
  is still mid-create fails on the snapshot; wait for create to settle (or for
  the instance to finish) before deleting. Delete the leftover snapshot
  afterward (`aws rds delete-db-snapshot`).
- **DynamoDB tables and the evidence + trace-archive S3 buckets are
  `DeletionPolicy: Retain`** to protect customer data, so a *failed* or
  rolled-back deploy leaves them behind. A naive retry then fails with
  `AlreadyExists`. Before retrying, delete the orphaned
  `<DdbTableNamePrefix>-*` tables and the `<ClusterName>-evidence-*` and
  `<ClusterName>-trace-archive-*` buckets (or recreate with a fresh
  `ClusterName` + `DdbTableNamePrefix`). The KMS CMK is also retained; schedule
  it for deletion (`aws kms schedule-key-deletion`, 7-day minimum window).
